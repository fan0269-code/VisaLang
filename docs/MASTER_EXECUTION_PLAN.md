# VisaLang 总体执行规划

> 版本：2026-07-13。本文是后续窗口的唯一执行顺序；每完成一个窗口，须在 `docs/TASK_LOG.md` 记录实际结果、遗留风险和下一窗口的前置条件。

## 文档权威顺序

1. 本文定义未来工作的唯一执行顺序和阶段准入条件。
2. `docs/OPERATIONS_STATUS.md` 记录当前线上、责任、隐私和商业化门禁；发生冲突时，以该文件的可核查状态为准。
3. `docs/PROJECT_CONTEXT.md` 是项目地图，不可单独作为上线、服务可用或业务审批的证据。
4. `docs/TASK_LOG.md` 是历史窗口记录；旧结论不得覆盖上述当前状态。
5. `docs/TRAFFIC_SITE_ROADMAP.md` 与 `docs/MONETIZATION_ROADMAP.md` 仅保留为战略参考，不能单独授权分析、留资、邮件、广告、支付或内容扩张。

## 1. 当前基线与总目标

技术与首轮产品建设已完成：Astro 路由、Germany A1/B1 决策路径、工具、中文 A1 核心路径、商业意图页、SEO/结构化数据和发布脚本均已存在。2026-07-12 本地复核 `npm test` 与 `npm run launch-check` 均通过；构建产出 98 个路由，发布检查 24/24 通过。生产站点在 2026-07-11 曾完成 200 状态验证，但每次发布仍须单独复查线上结果。

接下来不再以“多做页面”为目标，而是建立可验证的增长闭环：**官方核验优先的内容 → 清晰的下一步 → 可观测行为 → 用数据决定内容和商业投入**。Germany A1 与 B1 仍是样板路线；其他国家/考试只在有需求证据后扩张。

## 2. 贯穿所有窗口的规则

- 以 `src/` 为实现真相源，禁止通过手改 `dist/` 或重写根目录 legacy 文件解决问题。
- 每个窗口只拥有明确文件范围；不顺带改动其他模块、事实正文或视觉系统。
- 费用、接受证明、政策、资格和时间表必须引用当前官方来源；无法确认时只提供核验动作。
- 未确认支付、邮件发送、人工受理、交付或法律服务前，页面只能展示 `coming soon` 或 `request access`，不得制造完成状态。
- 每次代码/内容窗口的最低验收为相关测试、`npm run build`、`npm run launch-check`；发布另加线上路由和版本核验。

## 3. 分阶段执行路线

### 阶段 0：运营基线与责任确认（技术基线已记录，责任未完成）

**目的**：让“谁看数据、谁发布、谁回复咨询、谁审核事实”可追溯，消除旧文档与当前实现的差异。

**工作项**：复核生产部署合同（源码 commit、构建产物、Nginx root、回滚方式）；更新过时的项目上下文；建立一页状态面板，记录 Search Console、分析工具、联系入口、隐私/同意要求和负责人；对首页、A1 hub、B1 hub、工具中心、指南库及中文入口做线上冒烟检查。

**当前状态（2026-07-13）**：部署目录、历史发布证据、回滚产物和公网冒烟已被记录；但负责人、审批与可检查运营证据仍未完成。因此阶段 0 的技术部分不需要重复执行，责任确认仍是阶段 1 的硬前置条件。

**交付与验收**：一份当前状态记录、部署/回滚步骤、责任表和线上检查结果。所有“已上线”结论须含检查时间、URL 和 HTTP/页面标题证据。没有完整的责任确认，不进入付费、留资或批量扩展。

### 阶段 1：可观测增长与真实下一步

**目的**：先获得用户需求信号，再决定内容扩张方向。

**前置确认（需要业务方决定）**：分析工具及账户权限、Search Console property、隐私/同意方案、咨询/等待名单的真实接收端与回复 SLA。未确认时仅记录链接点击，不采集个人信息。

**工作项**：为路由选择、工具完成、官方来源外链、指南 CTA、联系意图建立最小事件表；提交并监控 sitemap；为 A1/B1/工具页统一一个可验证的下一步。事件只记录必要字段，不把工具结果或个人信息发送到未确认的服务。

**验收**：可从分析后台回答“用户从哪里进入、选择哪条路线、在哪一步离开、哪些页面带来 CTA”；事件命名和隐私说明有文档；上线后连续观察至少 4 周。

### 阶段 2：核心路线内容质量周期

**目的**：把已有 A1/B1 从“页面齐全”提升为“每一步可核验、可继续行动”。

**范围顺序**：先审计 Germany A1 的短页和高风险事实（考试中心、费用、补考、材料、豁免）；再审计 B1 永居/入籍页，把语言证明与其他资格条件严格拆开；最后只补充中文 A1 用户最常进入且确有缺口的页面。每次先建立“搜索意图 → 现有 slug → 官方来源 → 缺口 → 处理动作”表，优先深化旧页，禁止制造相似薄页。

**验收**：每篇变更页有明确适用边界、官方核验动作、更新时间和到 hub/工具/下一步的链接；不产生重复 slug、失效内链或无来源结论；通过全部发布门禁。

### 阶段 3：基于数据选择第二增长路线

**触发条件**：阶段 1 已积累至少 4 周数据，且能比较 A1、B1、UK SELT、Canada TEF/TCF 等入口的展示、点击、工具完成和 CTA 意图。

**决策规则**：优先选“已有自然需求 + 官方来源稳定 + 可形成 6–10 页完整决策路径”的路线，而非仅按题材热度。若数据不足，继续改善 A1/B1 标题、内链和下一步，不启动第二路线。

**工作项**：为唯一选定路线制作 hub、需求/可接受证明、考试比较、时间线、清单、官方来源与一至两个备考支持页；同时建立关键词—URL—CTA—schema 矩阵。

**验收**：新路线可以完成“确认要求 → 比较证明 → 清单/时间线 → 官方核验/联系意图”的闭环；新增页面均有独立意图和来源；在下一次月度复盘中与 A1/B1 按同一指标比较。

### 阶段 4：转化与商业化验证

**目的**：只把已确认能履约的服务转为真实产品，先验证需求，再扩大投入。

**前置确认（需要业务方决定）**：产品内容与交付方式、价格/退款规则、付款提供方、咨询接收流程、回复时限、数据处理责任，以及何时需要专业法律审阅。

**工作项**：先以小范围的 request-access 或人工确认验证 A1 Pack / Route Review 需求；仅在履约、支付与客服流程经过测试后开放真实购买或表单；审查广告、联盟和商业 CTA，确保它们不遮挡官方核验信息，也不暗示官方背书。

**验收**：每个 CTA 都有真实、可测试的目标和失败处理；用户可区分免费工具、资料包和信息性人工服务；转化率、人工处理量和投诉/退款情况进入月度复盘。

### 阶段 5：工程与发布常态化

**目的**：随着内容和流量增长，保持可重复、可回滚的发布质量。

**工作项**：将 `npm test`、构建和 launch check 固化为合并/发布门禁；每次发布记录 commit、构建时间、线上版本和回滚点；按月复核官方外链与站点健康；在确有维护负担时，再单独评估加入 lint/typecheck 或 CI，不为“以后可能有用”提前引入复杂工具。

**验收**：任一发布可在固定步骤内复现、验证和回滚；线上问题能定位到版本；内容、数据、商业和部署状态在同一周度记录中可见。

## 4. 固定节奏与下一动作

每周进行一次 30 分钟复盘：检查 Search Console/分析数据、官方来源变更、内容风险、CTA 状态和发布健康；每月只做一次“是否扩展路线/是否开放商业能力”的决策。没有数据或业务确认时，默认继续阶段 2 的官方来源刷新和核心页深化。

**下一执行窗口为“阶段 1 准入确认（无代码）”**：业务方须逐项在 `docs/OPERATIONS_STATUS.md` 补齐 Search Console、分析与数据保留、隐私/同意、联系接收与 SLA、事实审核、发布/回滚负责人及 SOP 的主备负责人和可检查证据。该窗口的验收仅有两种结果：全部确认后另开最小事件窗口，或保持阶段 1 关闭。不得新增页面、修改工具逻辑、接入或扩大任何第三方服务。
